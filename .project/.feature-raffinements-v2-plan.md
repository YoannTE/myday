# Feature plan — raffinements-v2

## Demande initiale

Lot d'améliorations produit après première utilisation en prod (9 points) :
dates optionnelles sur tâches, brief du jour réordonné, notes cliquables au
cockpit, heure de fin des événements, boutons « + » rapides par section, vues
planning jour/semaine/mois/année, sous-catégories de tâches personnalisables,
largeur de la boîte de note corrigée (mobile), section « Ta journée » →
« Ton planning » avec les 10 prochains événements.

## Besoin fonctionnel

Rendre le cockpit et le planning plus actionnables au quotidien : capture rapide,
navigation planning complète, organisation des tâches par catégorie, et un brief
qui suit l'ordre de lecture naturel (agenda → tâches → mails).

## État du code (constaté)

- `tasks.echeance` (timestamp nullable) existe déjà → date de tâche = UI + logique, **pas de migration**.
- `events.debut` ET `events.fin` existent déjà → heure de fin = **affichage UI**, pas de migration.
- Planning : `planning-jour.tsx` + `planning-semaine.tsx` existent → ajouter mois + année.
- Brief : service backend `backend/app/services/daily_brief/` (compose.py, context.py, degraded.py).
- Seule vraie migration : **catégories de tâches personnalisables** (nouvelle table + FK sur tasks).

## Découpage : 3 rounds

---

### Round 012 — Tâches : dates & catégories

**Features**

- [ ] F1 : Date d'échéance optionnelle éditable dans le formulaire de tâche (utilise `tasks.echeance` existant). Affichage de la date sur la carte de tâche (cockpit + page tâches). Les tâches avec échéance = aujourd'hui remontent dans le brief (préparé pour Lot 3).
- [ ] F2 : Catégories de tâches **personnalisables** — nouvelle entité `task_categories` (par utilisateur, RLS), CRUD (créer / renommer / supprimer), couleur optionnelle. Champ `categorieId` (FK nullable) sur `tasks`. Filtre/groupement par catégorie sur la page tâches et regroupement au cockpit.

**Impact existant**
- Tables modifiées : `tasks` (+ `categorie_id uuid` nullable, FK vers `task_categories`, ON DELETE SET NULL).
- Nouvelles entités : `task_categories` (id, user_id, nom, couleur?, createdAt, updatedAt) + RLS `user_id = current_setting('app.current_user_id')`.
- Pages modifiées : `/taches`, cockpit (taches-checklist.tsx).
- Nouveaux endpoints : CRUD catégories (FastAPI `POST/GET/PATCH/DELETE /api/task-categories`), MAJ endpoints tâches pour `echeance` + `categorieId`.

**Approche technique**
- Migration Drizzle `0008` : table `task_categories` + colonne `tasks.categorie_id` + RLS + grants `app_rls`. Suivre SOP RLS existante (policy texte, pas `::uuid`).
- Backend : service `task_categories`, endpoints protégés `Depends(get_current_user)`, pool `scoped_connection(user_id)`.
- Frontend : formulaire tâche (échéance via date picker shadcn, catégorie via select + création inline), affichage catégorie (badge couleur) sur les cartes.

**Dépendances** : aucune. **Packages** : aucun (date picker shadcn déjà dispo).

**Parcours** : l'utilisateur crée une tâche, y met une date + choisit/crée une catégorie « Pro ». Sur `/taches`, il filtre par catégorie. Au cockpit, ses tâches sont regroupées par catégorie.

**Permissions** : utilisateur authentifié, propriétaire uniquement (RLS). Pas de partage.

**Tests fin de round** : migration applique proprement (RLS active) ; créer/renommer/supprimer une catégorie ; assigner une catégorie + une date à une tâche ; supprimer une catégorie → tâches gardées, `categorie_id` remis à NULL ; cloisonnement (un user ne voit pas les catégories d'un autre) ; build + tsc + pytest.

### Round 013 — Planning : vues jour/semaine/mois/année + heure de fin

**Features**

- [ ] F3 : Heure de **fin** visible sur chaque événement (carte + détail), au format `14:00 – 15:30`. Le formulaire d'événement capture déjà début/fin ; s'assurer que la fin est bien éditée et affichée partout.
- [ ] F4 : Sélecteur de vue **Jour / Semaine / Mois / Année** dans l'en-tête du planning. Réutiliser `planning-jour` + `planning-semaine` existants ; ajouter `planning-mois` (grille calendaire) et `planning-annee` (12 mini-mois / densité). Navigation précédent/suivant adaptée à la vue. Mémoriser la vue choisie (préférence locale).

**Impact existant**
- Tables : aucune (events.fin existe).
- Pages modifiées : `/planning` (planning-client, planning-header).
- Nouveaux composants : `planning-mois.tsx`, `planning-annee.tsx`.
- Endpoints : `GET /api/events` doit accepter une fenêtre (from/to) selon la vue (jour/semaine/mois/année) pour ne pas tout charger.

**Approche technique**
- Frontend : état `vue` dans planning-client, fenêtre de dates calculée dans `date-utils.ts` (borne selon vue + fuseau `Europe/Paris`). Rendu mois = grille 7×(4-6). Année = 12 cellules avec compteur/points d'événements, clic → vue mois.
- Backend : endpoint events paramétré `from`/`to` (ISO) ; index `events_debut_idx` déjà présent.
- Affichage fin : `event-card.tsx` + détail → `HH:mm – HH:mm`.

**Dépendances** : aucune. **Packages** : aucun (pas de lib calendrier lourde ; grille maison Tailwind).

**Parcours** : l'utilisateur ouvre Planning, bascule Jour→Semaine→Mois→Année, navigue dans les mois, voit début+fin sur chaque événement. En vue année, clic sur un mois → ouvre ce mois.

**Permissions** : utilisateur authentifié, ses événements uniquement (RLS).

**Tests fin de round** : les 4 vues s'affichent, navigation prev/next correcte par vue, fenêtre de chargement correcte (mois ne charge pas l'année entière), heure de fin affichée partout, responsive mobile, build + tsc.

### Round 014 — Cockpit & brief repensés

**Features**

- [ ] F5 : **Brief du jour réordonné** — (a) rendez-vous du jour (TOUTE la journée courante), (b) tâches du jour (échéance = aujourd'hui), (c) 3 mails les plus importants de la boîte reçus dans les 5 derniers jours. Modifier le service `daily_brief` (compose.py + context.py) et l'affichage `brief-hero.tsx`.
- [ ] F6 : **Notes cliquables** depuis le cockpit — clic sur une note (notes-epinglees.tsx) ouvre son détail (`/notes` avec note ouverte).
- [ ] F7 : **Bouton rond bleu « + »** à côté du titre de chaque section du cockpit (Notes, Planning/Ton planning, Tâches) → ouvre la création rapide correspondante (dialog quick-add déjà existants pour notes/tâches ; event-form-dialog pour planning).
- [ ] F8 : Section « Ta journée » **renommée « Ton planning »** + affiche les **10 prochains événements à venir** (pas seulement aujourd'hui). `journee-timeline.tsx` → requête « à venir, limit 10 ».
- [ ] F9 : **Détail de note** — limiter la largeur/marge de la boîte de contenu à la colonne de base (max-w cohérent), pour que le texte ne déborde pas à droite (lisibilité mobile). Fix CSS dans `note-ouverte.tsx`.

**Impact existant**
- Tables : aucune.
- Pages modifiées : cockpit (brief-hero, notes-epinglees, taches-checklist, journee-timeline, cockpit-client), notes (note-ouverte).
- Endpoints modifiés : brief (contenu réordonné), events cockpit (10 prochains), pas de nouvel endpoint majeur.

**Approche technique**
- Backend brief : `context.py` récupère événements du jour (toute la journée, bornes `date_trunc('day' ... Europe/Paris)`), tâches échéance=aujourd'hui, top 3 mails (score, reçus ≤ 5 j). `compose.py` produit les 3 blocs dans l'ordre. `degraded.py` (mode règles) suit le même ordre.
- Frontend : brief-hero rend les 3 blocs ordonnés ; boutons « + » (composant réutilisable) branchés sur les dialogs existants ; notes-epinglees rend des liens ; journee-timeline requête « upcoming 10 » ; fix largeur note.

**Dépendances** : dépend du Round 012 (dates de tâches pour « tâches du jour ») et bénéficie du Round 013 (heure de fin dans « Ton planning »).

**Parcours** : au réveil, l'utilisateur ouvre le cockpit : brief = agenda du jour → tâches du jour → 3 mails clés. Il clique une note pour la lire. Il ajoute vite une tâche via le « + ». « Ton planning » liste ses 10 prochains rendez-vous.

**Permissions** : utilisateur authentifié, ses données (RLS + cloisonnement mails/brief déjà en place).

**Tests fin de round** : brief affiche les 3 blocs dans le bon ordre (avec données réelles et en mode dégradé) ; note cliquable ouvre le détail ; chaque « + » ouvre la bonne création ; « Ton planning » liste 10 événements à venir ; boîte de note lisible en mobile ; build + tsc + pytest.

## Corrections review (intégrées)

**Round 012 — Tâches**
- Couleur de catégorie **obligatoire**, auto-assignée depuis une palette à la création (l'utilisateur ne choisit pas forcément).
- Contrainte `UNIQUE (user_id, nom)` sur `task_categories` (évite les doublons au double-submit de la création inline).
- Migration `0008` : `GRANT SELECT,INSERT,UPDATE,DELETE ON "task_categories" TO app_rls` explicite ; index `task_categories_user_id_idx` + `tasks_categorie_id_idx` ; colonnes `createdAt/updatedAt` en `timestamptz DEFAULT now()`. SQL RLS **appendu dans le fichier de migration journalisé**, pas de `.sql` orphelin.
- **Cloisonnement FK** : la vérif de clé étrangère Postgres contourne la RLS → valider côté **service** que la catégorie appartient au user courant avant d'assigner `tasks.categorie_id`.
- États vides : aucune catégorie → tâches à plat + CTA discret « Créer une catégorie » ; les tâches `categorie_id NULL` → groupe « Sans catégorie » affiché **en dernier**, jamais masqué.

**Round 013 — Planning**
- **Fuseau `Europe/Paris` forcé partout** (bornes jour/semaine/mois/année) — corriger `date-utils.ts` qui calcule en heure navigateur ; aligné sur le brief (serveur UTC).
- Filtre événements en **chevauchement** : `debut <= :to AND fin >= :from` (garde les événements multi-jours/chevauchant la borne), pas un `BETWEEN` sur `debut`.
- Vue **année** : agrégat serveur (`GROUP BY jour`, COUNT) au lieu de charger tous les événements — endpoint/param dédié.
- Index composite `(user_id, debut)` pour mois/année (plus sélectif).
- Affichage fin : gérer `fin` un autre jour (multi-jours) et le cas `fin < debut` (pas de CHECK DB).
- Tests : **pytest** sur la fenêtre/chevauchement + multi-jours (pas seulement build/tsc).

**Round 014 — Cockpit & brief**
- Ordre des 3 blocs centralisé dans `compose.py` (source unique) ; `degraded.py` réutilise le même ordre. Prévoir un moyen de **forcer le mode dégradé** (clé IA absente) pour le qa-tester.
- Blocs vides du brief explicites : pas de RDV → « Aucun rendez-vous aujourd'hui » ; pas de tâche du jour → ligne omise ou « Rien d'urgent » ; moins de 3 mails → afficher 0 à 3 (pas de bloc fantôme). Tester en mode réel **et** dégradé.
- « Tâches du jour » (F5) bornées en `Europe/Paris`, pas une date UTC nue (décalage en soirée sinon).
- « Ton planning » (F8) : état vide « Aucun rendez-vous prévu » + bouton « + ».
- Après une capture rapide via « + » : petit **toast de confirmation** (l'utilisateur doit voir que ça a marché).
- Confirmer : le « + » de « Ton planning » ouvre `event-form-dialog` (événement), celui de Notes/Tâches la bonne création rapide.

**Hors périmètre (signalé pour un futur lot)** : tâches récurrentes, réordonnancement manuel des catégories.

## Ordre & déploiement

Ordre : **012 → 013 → 014** (le brief du 014 s'appuie sur les dates de tâches du 012 et l'heure de fin du 013). Après chaque round validé : push GitHub + redéploiement Dokploy (les apps sont déjà branchées sur le dépôt).
